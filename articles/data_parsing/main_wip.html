    Long(ish) Intro + Expand


    Background


    Computer Aided Engineering, while may seem a broad catergory by name, specifically refers to a set of tools used by the mechanical/aerospace/thermodynamics engineering to analyse mechanical systems. The analysis is a mathematical prediction of how a system will behave, in the time, frequency, and design domains.

    Early use of CAE involved hand written text files (even punch cards in early days) which define the properties and connectivity of the mechanical components, boundary conditions, and output requests, compromising the system, event, and output data of interest. These input decks are submitted to the mathematical solver which returns raw numeric output describing the various output requests defined in the input deck.

    Of course, over time software has developed to automate the generation of these inupt decks (pre-processors) and analysis of the output data (post-processors). The offer provide wizards for component generation, visualization of the model and animations of the analysis, and many other tools for model and results exploration.

    As opposed to the relatively simple structure of solver input decks pre/post processing software typically uses a more complex, hierarchal database structure to support higher level parametrization, configuration, and visualization definitions.

    Usually there is some support for programmatically interacting with these datasets via an Application Program Interface (API). Commands may be typed individually into a prompt, or stored in macros. The language of the commands may proprietary or some standard programming language (often Python) may be adapted , in which case a custom module for accessing the programs internal data set is provided.

    While the primary save files for these models maybe binary or ASCII, the many data files that it references are almost always ASCII. While many vendors have adopted standardized ASCII formats like XML, many files are proprietary, task-specific human readable ASCII formats.



    The need for customizaion


    The pre and post processing software typically are combined with the solver in a single suite from a vendor. In many way this makes sense, however, there are usually short comings when trying to develope unique, novel, or multi-disciplanary process. In this case, custom code must be generated. The code may interact with the pro/post processor via its API, the solver via subroutines, or with external datafile that serve as the go-between in the multi-stage process.

    Python for CAE


    Over the past 5 year, Python scriping and utility development has become an increasing dominant method for creating this customizaion at all stages, slowly replacing a myrid assorment of proprietary or obtuse scripting/programming languages. The broad adoption of an user friendly yet poweful scripting/programmnig language has dramitically improved the capabiliy of (non-software) engineers to develope improved CAE processes.

    Short Intro

    The engineering and scientific community use analysis software to model their resprctive systems. The software often relies heavily on many types datafiles. There has been a movement, in varying level of success, to move to standardize document formatx like XML or JSON, however the vast majorty still use task specific ASCII formats. Various needs arise to process these datafiles in a programatic manner. Python provides an excellent platform to do this.
